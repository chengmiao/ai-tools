# Orchestrator 指挥官模式

## 模型配置（在此修改）

| 角色 | 模型 | 如何修改 |
|------|------|----------|
| **Orchestrator（指挥官）** | 当前对话所选模型（建议 fable-5 / opus） | 用 `/model` 命令切换对话模型 |
| **Worker（打工人）** | `claude-sonnet-5` | 修改 `.claude/agents/worker.md` 中 frontmatter 的 `model` 字段 |

Worker 的模型定义在自定义子 Agent 文件 `.claude/agents/worker.md` 中，调用时通过 `subagent_type="worker"` 使用。

**回退规则**：如果 `subagent_type="worker"` 不可用，改用 `Agent(model="sonnet", subagent_type="Explore")`（只读搜集）或 `Agent(model="sonnet")`（需要执行时）。

**与 advisor 模式的关系**（互为镜像，按任务形态选用）：

| | advisor 顾问模式 | orchestrator 指挥官模式 |
|---|---|---|
| 主对话 | 便宜模型干活 | 贵模型指挥 |
| 子 Agent | 贵模型出主意（1 个，串行） | 便宜模型干活（N 个，并行） |
| 适合 | 深度编码任务，偶尔需要高人指点 | token 密集型任务，需要海量阅读/搜集 |

---

## 角色定义

**你（当前对话）= Orchestrator**，只做三件事：**拆解任务、分配工作、汇总结果**。

**Worker = 便宜模型子智能体**，每个 Worker 在自己的独立上下文里完成一个子任务，干完向你汇报压缩后的结果，然后退出。

核心纪律：**指挥官不亲自搬砖**。凡是需要大量读文件、抓网页、翻日志的活，一律派 Worker；你的上下文只装计划、Worker 的汇报和最终产出。

---

## 触发条件（满足任一即触发）

1. **token 密集型任务** — 大规模信息收集、多源调研、文档审查、日志分析、代码库普查
2. **可并行拆分** — 任务能拆成 ≥ 3 个互相独立的子任务（按文件/模块/来源/时间段切分）
3. **用户明说** — 「指挥官模式」「并行处理」「fan out」「批量分析」

单文件小改动、需要连贯上下文的深度推理任务**不适用**，直接自己做（或用 advisor 模式）。

---

## 执行流程

### 第 1 步：拆解（Plan）

- 把任务切成 **互相独立、边界清晰** 的子任务，每个子任务一个 Worker
- 切分维度：按文件/目录、按模块、按信息来源、按时间段
- 子任务数量参考：3-8 个为宜；超过 10 个时分批派发

### 第 2 步：派发（Fan out）

**关键：互相独立的 Worker 必须在同一条消息里并行调用**（多个 Agent 工具调用放在一个块里）。耗时很长的子任务可加 `run_in_background=true`。

每个 Worker 的 prompt 必须自包含（Worker 看不到你的对话历史）：

```
Agent(
    subagent_type="worker",
    description="Worker: {子任务简称}",
    prompt="""## 子任务
{这一个 Worker 要做的具体事情，边界清晰}

## 上下文
{必要的背景：项目路径、目标格式、约束条件 —— Worker 看不到主对话，缺什么补什么}

## 输出要求
按以下格式返回，总量 ≤ 500 token：
{明确的结构，如：
- 每条发现：文件路径:行号 + 一句话结论
- 或：JSON 列表，字段为 xxx}
找不到时如实返回「未找到 + 已尝试的途径」。
"""
)
```

### 第 3 步：汇总（Synthesize）

1. 收齐所有 Worker 汇报后再综合，**交叉比对、去重、解决矛盾**
2. Worker 汇报有矛盾或明显缺口时，**补派一轮 Worker** 定向核实（同一缺口最多补 1 轮）
3. 用指挥官（贵模型）的能力产出最终结果：这是贵模型该花 token 的地方
4. 向用户交付时注明信息来源（哪个 Worker / 哪个文件 / 哪个网页）

---

## 派发纪律

- **独立性**：子任务之间不允许有依赖；有依赖就分两轮派发（第一轮结果决定第二轮任务）
- **输出限额**：每个 Worker 的 prompt 里必须写明输出格式和 token 上限，防止 Worker 把原始材料整段倒回来
- **不重复劳动**：派出去的活自己不要再做一遍；Worker 失败/超时才接手或重派
- **只读优先**：纯搜集类子任务用只读 Worker（回退时选 `Explore`），避免误改文件

---

## 成本说明

- 90%+ 的 token 消耗在 Worker（Sonnet 价格）：海量阅读、搜索、抓取
- Orchestrator（Fable 5 价格）只消耗：任务拆解 + 各 Worker 的压缩汇报 + 最终汇总
- 与全程用贵模型相比：结果质量接近，成本大幅下降；与全程用便宜模型相比：汇总质量显著更高
