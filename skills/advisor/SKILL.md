# Advisor 顾问模式

## 模型配置（在此修改）

| 角色 | 模型 | 如何修改 |
|------|------|----------|
| **Executor** | 当前对话所选模型（建议 sonnet） | 用 `/model` 命令切换对话模型 |
| **Advisor** | `claude-fable-5` | 修改 `.claude/agents/advisor.md` 中 frontmatter 的 `model` 字段 |

Advisor 的模型定义在自定义子 Agent 文件 `.claude/agents/advisor.md` 中，调用时通过 `subagent_type="advisor"` 使用，**无需手动切换对话模型**。

**回退规则**：如果 `subagent_type="advisor"` 不可用或模型名被拒绝，改用 `Agent(model="opus", ...)`（当前最强别名）。

---

## 角色定义

**你（当前对话）= Executor**，负责所有实际工作：写代码、读文件、跑命令、改 bug。

**Advisor = 上方配置的 ADVISOR_MODEL（默认 `claude-fable-5`）**，只在三个关键节点出场，通过 Agent 工具调用，输出 400-700 token 的战略建议后退出。

---

## 三个出场节点

| 节点 | 时机 | Advisor 职责 |
|------|------|--------------|
| **① 开局定方向** | 非平凡任务动手之前 | 给出技术路线、拆解步骤、指出风险点 |
| **② 中途解困** | 连续失败 ≥ 3 次，或遇到架构决策 | 诊断根因、推荐路径 |
| **③ 收尾把关** | 宣布任务完成之前 | 审查改动，判定放行 / 需返工 |

### 节点 ① 开局定方向（触发条件）

任务满足任一条件时，**先咨询再动手**：
- 预计涉及 3 个以上文件的改动
- 有多种可行方案需要选型
- 新功能 / 重构 / 涉及对外接口或数据结构的变更

琐碎任务（改错别字、单行修复、回答问题）直接做，不触发。

### 节点 ② 中途解困（触发条件）

1. **连续失败 ≥ 3 次** — 同一问题用不同方法尝试了 3 次仍失败
2. **架构决策关键词** — 对话中出现以下词语时主动提议咨询：
   - 架构、重构、设计模式、方案选型、trade-off
   - 如何设计、哪种方案更好、最佳实践
   - 数据库设计、API 设计、模块拆分、性能瓶颈

### 节点 ③ 收尾把关（触发条件）

凡是触发过节点 ① 的任务，完成后**必须**过一次把关；其他任务在用户说「把关」「验收」「review 一下」时触发。

## 调用模板

统一使用 Agent 工具调用 Advisor 子 Agent（模型已在 `.claude/agents/advisor.md` 中固定为 fable-5），按节点选用对应 prompt：

### ① 开局定方向

```
Agent(
    subagent_type="advisor",
    description="Advisor 开局定方向",
    prompt="""你是战略顾问。Executor 即将开始一个任务，请给出方向性指导。

## 任务描述
{用户的原始需求 + 你对任务的理解}

## 相关背景
{项目技术栈、相关文件路径、已知约束}

## 候选方案（如有）
{你初步想到的 1-3 种做法}

请给出 400-700 token 的开局指导：
- 推荐的技术路线（具体到方案/API/模式，说明为什么优于其他候选）
- 建议的实施步骤拆解（3-6 步）
- 最容易踩坑的 1-2 个风险点及规避方式
- 不需要写实现代码，Executor 负责执行
"""
)
```

### ② 中途解困

```
Agent(
    subagent_type="advisor",
    description="Advisor 战略咨询",
    prompt="""你是战略顾问。Executor 遇到了问题需要你的指导。

## 当前任务
{当前任务的简要描述}

## 已尝试的方案
{列出已经尝试过的方法和失败原因}

## 需要建议的问题
{具体问题}

请给出 400-700 token 的战略建议：
- 诊断根本原因
- 推荐最佳路径（具体到使用什么方案/API/模式）
- 不需要写实现代码，Executor 负责执行
"""
)
```

### ③ 收尾把关

```
Agent(
    subagent_type="advisor",
    description="Advisor 收尾把关",
    prompt="""你是验收顾问。Executor 认为任务已完成，请你把关。

## 任务目标
{原始需求 + 开局时 Advisor 给的方向（如有）}

## 完成情况
{改了哪些文件（给出路径）、做了哪些验证（测试/运行结果）}

## 自查存疑点（如有）
{Executor 自己不确定的地方}

请用 Read/Grep 抽查关键改动后，给出 400-700 token 的验收结论：
- 明确判定：**放行** 或 **需返工**
- 若需返工：列出具体问题（按严重程度排序），指出问题所在文件
- 检查点：是否偏离既定方向、是否有遗漏的需求点、是否引入明显隐患（边界条件/错误处理/向后兼容）
- 不需要写修复代码，Executor 负责整改
"""
)
```

---

## 收到建议后

1. **开局建议**：向用户简述 Advisor 给的方向，然后按该路线执行；如与用户意图冲突，以用户为准
2. **解困建议**：作为上下文继续工作，同一问题只咨询一次；解决则继续，仍有疑问可再次触发
3. **把关结论**：
   - **放行** → 向用户汇报完成，并附上 Advisor 的验收要点
   - **需返工** → 逐条整改后再过一次把关（同一任务把关最多 2 轮，第 2 轮仍不通过则把分歧如实呈给用户裁决）

## 成本说明

- 90%+ 的 token 消耗在 Executor（Sonnet 价格）
- Advisor 每次出场约 700 token（Fable 5 价格）
- 每个任务通常调用 2-3 次 Advisor（开局 1 次 + 把关 1 次，卡住时再加 1 次）
- 效果约等于 92% 的 Fable 5 能力，花费约 63% 的成本
